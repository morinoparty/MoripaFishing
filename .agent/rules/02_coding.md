# コーディングプラクティス

## 実装方針

1. まず既存コードの責務分離と命名規則を確認します。
2. 変更対象が `api` なのか `bukkit` なのか `integrations` なのかを明確にします。
3. 既存の型と公開 API を壊さない範囲で、小さく実装します。
4. 必要に応じてテストとドキュメントを更新します。

## Kotlin / Minecraft 側の原則

- `api` には Paper/Bukkit 実装詳細を持ち込まないでください。
- `bukkit` には Paper 実装、コマンド、イベント、設定読み込み、ゲーム内挙動を配置してください。
- `integrations` は追加連携やオプショナル機能を切り出す場所です。
- 既存コードが `KoinComponent` と `by inject()` を広く使っているため、周辺実装に合わせて統一してください。
- Arrow は既存コードで利用されている箇所に合わせて使い、不要な箇所へ無理に導入しないでください。
- Adventure / MiniMessage を使っている箇所では、既存のメッセージ表現を優先してください。

## コードスタイル

- 常に既存コードの設計・記法・パッケージ分割を優先してください。
- 1 ファイル 1 クラスを原則とします。
- 過度な抽象化や将来前提の汎化は避けてください。
- コメントは必要な箇所にだけ簡潔に入れてください。各行コメントを前提にしないでください。
- KDoc は公開 API や意図が伝わりにくい箇所に限定して追加してください。
- 新規コードは ASCII を基本とし、既存ファイルに合わせる場合のみ例外とします。

## 確認

- Kotlin 側を変更したら、まず `task test`、必要に応じて `task check` や `task build` を実行してください。
- `task check` は `ktlintFormat`、`detekt`、`test` を順に実行します。
- エディタ上の一時的な unresolved 表示は、実ビルドと矛盾しない限り過信しないでください。

MoripaFishing は **Integration** と **Addon** の 2 種類の外部プラグインと連携します。
依存の向きが逆になるため、役割も異なります。

| 区分 | 依存関係 | 代表例 | 特徴 |
|---|---|---|---|
| **Core** | — | `MoripaFishing` 本体 | 魚抽選・釣りイベント・天候判定などの最小機能 |
| **Integration** | Core が Integration を参照 | `MoripaFishing-Integration-WorldLifecycle` | Core が SPI 経由で機能の実体を借りる。未導入時は該当機能が無効 |
| **Addon** | Addon が Core をソフト依存 | `MoripaFishing-Addon-CatchAnnounce` | Core の公開 API とイベントを利用して拡張する外部プラグイン |

```mermaid
flowchart BT
    core[MoripaFishing-Core]

    int1[Integration 1]
    int2[Integration 2]

    addon1[Addon 1]
    addon2[Addon 2]

    core -.optional Load.BEFORE.-> int1
    core -.optional Load.BEFORE.-> int2

    addon1 -.soft depend.-> core
    addon2 -.soft depend.-> core
```

Addon 側も Core をハード依存にはしません。`server("MoripaFishing", Load.BEFORE, required = false)`
として宣言し、Core が導入されていないサーバーでは Addon が起動時に自身を無効化します。

## Integration の仕組み

Integration は独立した Bukkit プラグイン jar として提供され、
以下のように Core から検出・利用されます。

1. Integration の `JavaPlugin` クラス自身が SPI インターフェースを実装する。
   SPI はコアの `:api` モジュールではなく、`integrations/<名前>-api` という独立した
   Gradle モジュール（`party.morino:moripafishing-world-lifecycle-api` の `WorldLifecycleProvider`、
   `party.morino:moripafishing-weather-api` の `WeatherControlProvider`）に置かれており、
   いずれもコアの `moripafishing-api` に依存しない。
2. Integration の `paper-plugin.yml` では、原則として MoripaFishing への読み込み順を課す
   依存（Paper の `Load.BEFORE` / `AFTER`）を**書かない**。
   Core 側が Integration を `Load.BEFORE` で参照するため、順序を課すと循環するためである。
   ただし Core の公開 API を実行時に参照する機能（例: WorldLifecycle の参加時テレポート）では、
   `join-classpath` 目的で `Load.OMIT`（順序制約なし・`required: false`）の依存を宣言してよい。
   旧 Bukkit ローダー向けのフォールバックとして `bukkit-plugin.yml` 側に `softdepend` を併記する場合もあるが、
   こちらは循環を許容するため順序制約の問題は起きない。
   実際、Weather Integration は Core の API を実行時に呼ばないため依存を一切宣言しておらず、
   `Load.OMIT` を宣言しているのは WorldLifecycle Integration だけである。
3. Core の `paper-plugin.yml` では Integration を
   `server("<Integration 名>", Load.BEFORE, required = false)` として宣言し、
   ロード順を「Integration → Core」に揃える。
4. Core の `onEnable` で `Bukkit.getPluginManager().getPlugin("<Integration 名>")`
   を取得し、SPI 型にキャストできれば採用する。
5. Integration 未導入時は Core が `null` を保持し、該当機能はスキップされる。
   プラグイン自体は導入されているのに SPI 型を実装していない場合は、
   Core が警告ログを出したうえで同じく `null` として扱う。

### SPI モジュールの扱い

Integration の SPI インターフェースは、次のように Core とも Integration 本体とも独立したモジュールになっています。

| SPI モジュール | インターフェース |
|---|---|
| `integrations/world-lifecycle-api`<br />(`party.morino:moripafishing-world-lifecycle-api`) | `party.morino.moripafishing.integrations.worldlifecycle.api.WorldLifecycleProvider` |
| `integrations/weather-api`<br />(`party.morino:moripafishing-weather-api`) | `party.morino.moripafishing.integrations.weather.api.WeatherControlProvider` |

Core はこれらを `compileOnly` で参照するだけで shade しません。Integration 側も同様に shade せず、
実際に実行時のクラスを供給するのは Integration プラグインの jar です。
Core は Paper の `join-classpath` を通じてそのクラスを共有し、同一の型として `is` 判定できるようになります。
Integration を自作する場合も、コアの `:api` ではなく該当する `-api` モジュールに `compileOnly` で依存してください。
これらの SPI モジュールは Maven Central に公開されているため、外部プロジェクトからは
`party.morino:moripafishing-world-lifecycle-api` / `party.morino:moripafishing-weather-api` の座標で解決できます。

:::info
Addon との違いは **「誰が誰を検出するか」** です。
Integration は Core が能動的に検出して取り込むのに対し、
Addon は Core のイベントを購読したり、`MoripaFishingAPI` の公開メソッドを呼び出したりして自分から Core に働きかけます。
Core が Addon 向けに提供している登録 API は `registerWeatherSource` / `unregisterWeatherSource`（天候ソースの差し替え）で、
公式 Addon の [CatchAnnounce](/docs/addons/catch-announce) のように Bukkit のイベントリスナー登録だけで完結する構成も一般的です。
:::

## 利用側の確認方法

`WorldLifecycleProvider` / `WeatherControlProvider` は Core が shade しない独立モジュールの型のため、
これらを返すメソッドは `MoripaFishingAPI` には公開されていません。
外部プラグインから確認する場合は、Integration プラグインを直接取得してキャストします。

```kotlin
val worldLifecycle =
    Bukkit.getPluginManager().getPlugin("MoripaFishing-Integration-WorldLifecycle") as? WorldLifecycleProvider
if (worldLifecycle == null) {
    logger.warning("WorldLifecycle Integration が未導入です")
}
```

## Integration 未導入時の挙動

Integration が無くても Core は起動し、欠けている機能だけが縮退します。
未導入時の Core API の挙動は各機能側で吸収されるため、呼び出し側が例外を受け取ることはありません。

- **WorldLifecycle 未導入**: ワールド境界の同期と、カスタムジェネレーターによるワールド作成が無効になります。
  `WorldManager.createWorld` は警告ログを出したうえで `false` を返し、`WorldManager.getGeneratorIds` は空リストを返します。
- **Weather 未導入**: 天候の決定や、天候を条件に含む魚の抽選は Core 側で動き続けます。
  スキップされるのは「決まった天候を実際のワールドへ適用する」部分だけです。

## 提供される Integration

現時点で公式に提供されている Integration は以下です。

- [`MoripaFishing-Integration-WorldLifecycle`](/docs/integration/world-lifecycle): ワールド境界の同期・カスタムジェネレーターでのワールド作成
- [`MoripaFishing-Integration-Weather`](/docs/integration/weather): 釣りワールドへの天候の適用（嵐の切り替えと CLOUDY のクライアント側バリア天井）

## 提供される Addon

公式に提供されている Addon は以下です。

- [`MoripaFishing-Addon-CatchAnnounce`](/docs/addons/catch-announce): 釣果のアナウンス

## 命名規則

プラグイン名からも種別が一目でわかるよう、外部プラグインの名前には共通の接頭辞を付けています。
Integration は `MoripaFishing-Integration-<名前>`、Addon は `MoripaFishing-Addon-<名前>` という形式で命名されます。
`plugins/` フォルダや依存プラグイン一覧に並んだときも、この接頭辞を見れば Core が能動的に検出する Integration なのか、
Core をソフト依存して自ら働きかける Addon なのかをすぐに判別できます。
